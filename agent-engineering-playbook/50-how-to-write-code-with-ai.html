How To Write Code With AI

A field guide to driving an AI coding agent through the playbook without reading every process file yourself. The short version: you own intent and approval; the agent owns execution, verification, and keeping the project documents current.

Status · Source of truth Last verified · 2026-06-05 · playbook V0.1.8 Audience · Human project owner Format · Interactive HTML (canonical)

Acknowledgement

This playbook grew out of an exploration of how to turn modern AI-agent skills into a practical engineering process for building real applications. It is especially influenced by Matt Pocock's work on composable engineering skills, including mattpocock/skills and skills.sh, and by Garry Tan's GStack project and garrytan/gstack. This guide adapts those ideas into a human-led, agent-assisted playbook built around explicit alignment, shared language, TDD, review, QA, shipping, and learning loops.

02The development loop

Five stages, run in order. Loop back to Align for the next feature. Keep the trips around the loop short — one slice at a time, with the agent pausing for your go-ahead after each slice.

01 · ALIGN

Use this when the feature is still fuzzy

Say:

I want to build [feature].

The agent should grill the plan, settle terms, and write the decisions into planning docs, , and where needed.

Mockup before code, for anything with a screen. Once the grilling settles the vocabulary, the agent should turn the agreed terms into an HTML — in the project's own visual language — and show it to you before any implementation planning. You react to a real screen instead of a description, which heads off layout churn later. Alignment isn't done until the mockup is accepted or you've explicitly deferred it.
02 · SLICE

Once the feature is understood, break it into small slices

Say:

Break this into slices.

Each should produce something testable. Avoid giant all-or-nothing builds.

03 · BUILD

Ask for one slice at a time

Say:

Build slice 1.

The agent should implement, test, and show what changed. If the app has a UI, the agent should use the browser to check the actual screen at desktop, tablet, and mobile widths — not just rely on code inspection. See §04 for the full it owes you.

Branch first. For any non-trivial product slice, bug fix, or , the agent should start from a fresh before coding. Moving finished work onto a branch at the end is only a recovery path when work accidentally started on main.
One slice, then a pause. When a slice is built and verified, the agent makes one for it and then stops — it will not roll straight into the next slice on its own. It reports what changed, which checks passed, and the commit, then asks whether to continue, pause, or change direction. Each slice boundary is a decision point that belongs to you. Reply Build the next slice. when you're happy to proceed.
04 · REVIEW & CORRECT

Use plain language for feedback

Say:

This label is wrong.
This screen should be desktop-only.
This breaks on my phone.
This workflow is missing a way to restore removed people.

The agent should convert that feedback into code, tests, and documentation updates. For UI work, its must report how the screen behaves at desktop, tablet, and mobile widths, so a phone-sized regression is caught here rather than after ship.

05 · SHIP

When the feature is complete

Say:

Ship the feature.

The agent should now follow the ship sequence — see §03 below.

The manual GitHub step

After the agent opens the pull request, you must go to GitHub yourself. Do this manually:

  1. Open the PR link the agent gives you.
  2. Read the PR summary.
  3. Skim the changed files.
  4. Check that passed, if exists.
  5. If it looks right, mark the PR ready if it is still draft.
  6. Merge the PR.
  7. Tell the agent:
I reviewed and merged the PR. Can you check?

This manual step matters because the human remains accountable for what enters main. The agent can prepare the PR, but it should not be treated as the final approver.

After merge

After you tell the agent the PR was merged, it should:

  1. Confirm the PR is merged.
  2. Fetch or otherwise verify the merged state.
  3. Check whether any post-merge step is needed.
  4. Recommend the next useful action.

If there is a deployment target, this is where deploy, , and work happens. If there is no deployment target yet, the next step is usually choosing the next feature or running a .

Before starting the next feature, the agent should switch back to main, pull the merged GitHub state, and create a new feature branch from there.

04The UI smoke floor

For anything with a screen, this is the minimum the agent must check before calling a slice done — at desktop, tablet, and mobile widths. You don't have to memorise it; just know you can hold the agent to it. Most of the late, annoying bugs in real projects were caught here or not at all.

  • No sideways scroll. The page body shouldn't overflow horizontally on a phone.
  • Tables stay readable. Wide data tables scroll inside their own box rather than crushing columns into nonsense.
  • Tappable things are big enough. Buttons, links, and chips meet the ~44px floor on mobile, unless a compact exception is written down.
  • Headings match the visual order. The underlying lines up with what the eye sees.
  • URL-driven views keep their place. Drawers, filters, tabs, and expanding rows that live in the should preserve your scroll position and keep focus where you expect — not jump you back to the top.
How to invoke it. You rarely need to. The agent owes you this floor automatically on UI work. If you want to be explicit, say Run the responsive smoke check. or just Check this on mobile too.

06Rules of thumb

If you only remember a handful of things from this guide, remember these.

  1. Keep decisions small and explicit.
  2. Insist on an HTML before building any UI that matters.
  3. Start non-trivial work on a feature branch, not on main.
  4. Review each slice as it lands — the agent now pauses and commits after every verified slice, so use that gap.
  5. Prefer one completed slice over several half-finished branches.
  6. Treat tests and the desktop/tablet/mobile browser check as part of the work, not an optional extra.
  7. Do not skip the GitHub review and merge step.
  8. Expect the agent to tidy and archive the docs at ship — a clean doc-close is part of shipping.
  9. Keep local databases, secrets, and generated build output out of git.
