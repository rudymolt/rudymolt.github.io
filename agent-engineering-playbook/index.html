Never written code? Two minutes of background

Eight words the guides lean on. Everything else is explained as you meet it.

Code
—
instructions that tell a computer exactly what to do. An app is thousands of lines of it.
Repo
—
the folder where a project's code and its full history live.
Database
—
where an app stores its information permanently, like spreadsheets the code can read and write.
Server
—
a computer running somewhere else that answers requests — book online, and a server records it.
UI
—
the part of the app you can see and touch: screens, buttons, forms.
Test
—
a small piece of code that automatically checks a bit of the app behaves as it should.
Deploy
—
putting a new version of the app live so real people can use it.
Production
—
the live version real users are using right now, as opposed to a private working copy.

What The AI Engineering Playbook Is

The AI Engineering Playbook is a way of working in which a person stays in charge while an AI coding agent does the building. It is not a piece of software or a replacement for engineering judgement. It is a way to run a coding session so the agent works inside a clear process: understand the task, agree the plan, break the work into small that can each be shown working on their own, build one slice at a time, review the result, test it, and ship only when the evidence is good enough.

How to brief the agent

Know what background, limits, signs of success, and proof to give the agent before it changes any code.

Process route map

Decide whether the next move is agreeing the plan, splitting the work up, building, reviewing, quality-checking, fixing a fault, shipping, or learning from how it went.

Verification discipline

Match the amount of checking to the risk, so small changes stay fast and important ones get tests, a check in the browser, a review, and a careful release.

Why It Exists

Claude, Codex, and similar coding agents can move very quickly, but speed creates a new problem: a vague request can lead to sprawling changes, a flimsy underlying design, no tests, and a result no one else can pick up later. The playbook exists to keep the person in charge of what to build and which trade-offs to make, while the agent handles the focused work of building it. It gives both sides a shared rhythm for writing code without turning every session into guesswork.

How To Use It With Claude Or Codex

  1. Read How to drive the agent first. It follows one feature from first sentence to shipped code, and teaches the vocabulary every other page uses.
  2. Use the process map when you are unsure what kind of request you are making. It points the work to the right next step: agreeing the plan, splitting the work up, building, reviewing, quality-checking, fixing a fault, shipping, or a look back at how it went.
  3. For new features, use the alignment guide before code and the breakdown checker before building. The goal is one small slice that works end to end and can be shown off on its own, rather than a sweeping rewrite.
  4. While the agent is building, keep it focused on one slice, ask it to spell out any risky assumptions, and insist on tests or browser checks that fit the risk of the change.
  5. Before accepting the work, use the foundations, review, quality-check, and document lifecycle pages to check whether the result is easy to understand, easy to change later, tested, and ready to hand over or publish.

What Was Published Here

This publication turns the playbook into linked, static HTML pages for people using coding agents. It includes this hub, the foundations guide, the 13-stage process map, focused alignment and breakdown explainers, the frontend-track guide, the document lifecycle guide, the practical agent-driving guide, the theory page that explains the reasoning behind the process, and a browsable glossary of every term the guides use.

Acknowledgements

This playbook builds on ideas and practices shared openly by others working on coding agents. In particular, it acknowledges Matt Pocock's work on practical coding-agent skills, including mattpocock/skills and skills.sh, and Garry Tan's GStack project and garrytan/gstack.

The pages here do not replace those projects. They organise selected ideas into a human-facing workflow for deciding what to ask an agent to do, when to pause for judgement, and how to verify the result.

This hub keeps the different kinds of page separate. The "drive the agent" page (§50) is a step-by-step how-to. The theory page (§60) explains the reasoning.
