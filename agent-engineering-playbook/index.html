What The AI Engineering Playbook Is

The AI Engineering Playbook is a human-led workflow for agent-assisted software development. It is not a framework, model, or replacement for engineering judgement. It is a way to run a coding session so the agent works inside a clear process: understand the task, agree the plan, break the work into vertical slices, implement one slice at a time, review the result, test it, and ship only when the evidence is good enough.

Prompting structure

Know what context, constraints, acceptance criteria, and evidence to give the agent before it edits code.

Process route map

Decide whether the next move is alignment, slicing, implementation, review, QA, debugging, shipping, or learning.

Verification discipline

Match checks to risk, so small changes stay fast and important changes get tests, browser QA, review, and rollout care.

Why It Exists

Claude, Codex, and similar coding agents can move very quickly, but speed creates a new problem: vague requests can produce broad changes, shallow architecture, missing tests, and unclear handoffs. The playbook exists to keep the human in charge of intent and tradeoffs while the agent handles focused execution. It gives both sides a shared rhythm for building code without turning every session into improvisation.

How To Use It With Claude Or Codex

  1. Start with the process map when you are unsure what kind of request you are making. It routes the work to alignment, breakdown, implementation, review, QA, debugging, shipping, or retro.
  2. Open How to drive the agent before a real coding session. Use it to shape the prompt you give Claude or Codex: what the goal is, what files matter, what constraints apply, and what evidence you expect back.
  3. For new features, use the alignment guide before code and the breakdown checker before implementation. The goal is one vertical, demoable slice rather than a broad rewrite.
  4. During implementation, keep the agent focused on one slice, ask it to explain risky assumptions, and require tests or browser checks that fit the risk of the change.
  5. Before accepting the work, use the foundations, review, QA, and document lifecycle pages to check whether the result is understandable, maintainable, tested, and ready to hand off or publish.

What Was Published Here

This publication turns the playbook into linked, static HTML pages for people using coding agents. It includes this hub, the foundations guide, the 13-stage process map, focused alignment and breakdown explainers, the document lifecycle guide, the practical agent-driving guide, and the theory page that explains the reasoning behind the process.
