Quickstart

One feature taken through all thirteen stages, so you can see what each stage actually produces before you try it on something real. Every item of work shown here is genuine and small.

The project

MeetTrack records athletics results (athlete, event, mark, meet date), ranks results tables for each event, and answers questions about personal bests. It is written in Python, has no on-screen interface, and is checked with an automated test command (python3 -m unittest). It was set up using this playbook — with CLAUDE.md (the agent's standing instructions, including the first rule), AGENTS.md, CONTEXT.md, a progress-tracking file, a schedule of regular checks, and planning/ and archive/ folders. One feature has already been released and filed away.

The request

Human: "Coaches keep asking if athletes are actually improving. Can the library show that?"

What follows is the full loop, stage by stage.

The thirteen stages

00 · Prereqs — every session, first

The agent checks that its tools are installed, the project files are present, the playbook version is current, the "no on-screen interface" setting is recorded (so the visual checks are skipped), and the existing tests all pass. Result: pass, in about thirty seconds. The stage's Next note then sends a feature request on to the agree-the-plan stage.

01 · Agree the plan — question hard before any code

The first rule kicks in: this is a feature request, so no code yet. The agent keeps asking questions until the shape is settled — the goal in one sentence ("a coach can see one athlete's progress in one event over time"), a history list rather than a ranking, oldest result first, and a clear list of what is being left out (charts, an on-screen interface, and saving anything to disk). The word "progression" is settled on as "history view" throughout, so the project doesn't end up with two words for the same thing.

02 · Context and decision records — write it down while it's fresh

CONTEXT.md gains the term history view — a list of one athlete's results in one event in date order, oldest first, and clearly not the same as the results table. No new decision record is needed (a decision record is a short note explaining a significant design choice): no real design choice was made here, because the earlier decision to keep all the data in memory still covers the one constraint that matters.

03 · Requirements doc — write down what was agreed

A short requirements document (about 25 lines) captures what's needed: a history(athlete, event) request returns the results in date order, earliest first; the personal best can be picked out; results on the same date keep the order they were entered in; and nothing changes about how data is saved. Success: a coach can answer "is Aisha improving in the 100m?" in one go. The progress-tracking file is updated to show the requirements are written.

04 · Breakdown — split into small, complete pieces

The work is split into three small pieces, each one a complete slice from data store to result to tests: (1) the store can return one athlete's results for one event in date order; (2) the history() request, with the personal best marked; (3) a small helper that formats the results for display. The progress-tracking file now shows three open pieces of work.

05 · Triage — pick what to do next

Piece 1 wins — everything else depends on it. On a busier project this stage also keeps the task list tidy: closing tasks that no longer matter and combining duplicates.

06 · Architecture — only when it's due

Not this time: not enough pieces of work have been completed since the last architecture review to make it due. The stage exists so the check happens, not so it runs every time.

07 · Building it — write the test first

Piece 1, the failing test first:

def test_history_is_chronological_oldest_first(self):
    rows = self.t.history("Aisha", "100m")
    self.assertEqual([r.meet_date for r in rows],
                     sorted(r.meet_date for r in rows))

Then the smallest amount of code that makes the test pass, then a tidy-up; the saved change is labelled following the project's usual naming pattern. The progress-tracking file now shows two open pieces of work, the architecture-review counter goes up by one, and the at-a-glance status summary is refreshed.

08 · Review — catch what the tests don't

One read-through before the change is proposed for merging into the main project: does it match the requirements document, does it stick to the agreed words ("history view", not "progression log"), and is the claim that same-date results keep their order actually guaranteed? Python's sorting does keep that order, and that fact is written into the change's description rather than quietly assumed.

09 · Quality check

With the "no on-screen interface" setting on and nothing relying on outside services, the verification ladder is happy for the passing automated tests to vouch for this piece — the stage is genuinely considered, not quietly skipped. The whole feature is released once all three pieces are done.

10 · Release — and tidy up the documents

After piece 3 is done, the feature is released and the document tidy-up happens now, when the whole feature ships — not after each piece: a release note is written, the planning folder is moved into archive/, both STATUS.md files are brought up to date, and the progress-tracking file is stamped as done.

11 · Fixing bugs — didn't happen this time

No bugs this time round. When one does turn up: work out the cause before fixing it, and treat the fix itself as another small piece back at stage 07 — writing a failing test first.

12 · Look back and learn

Ten minutes, answered honestly. Went well: the hard questioning caught the "ranking versus history" mix-up before any code was written. Didn't: piece 1 nearly slipped in some unrelated tidying-up, which was spotted and separated out at review. Lesson kept: "keep tidy-up changes separate from new-feature work" was added as a standing rule in CLAUDE.md.

What to take from this

  • The documents are small. A 25-line requirements doc, a three-row table of work, a six-line look-back review. The discipline is in the order you do things, not in the paperwork.
  • The agent never chose the route itself. The stage and the signposts at the end of each stage did — that's the V0.2 design making the path easy to follow.
  • Three stages did nothing this time (architecture, bug-fixing, and in-browser testing) and that's correct: they're checks that can happen when needed, not rituals that must run every time.
This example is measured, not wishful. A part-finished version of this exact project is kept as a fixed test case, used to measure how well AI agents find their way through it.
