The project

MeetTrack records athletics results (athlete, event, mark, meet date), ranks results tables per event, and answers personal bests. Pure Python, no UI, tests via python3 -m unittest. It was bootstrapped from this playbook — CLAUDE.md (with the first rule), AGENTS.md, CONTEXT.md, a state file, cadences, and planning/ + archive/ folders. One feature has already shipped and been archived.

The request

Human: "Coaches keep asking if athletes are actually improving. Can the library show that?"

What follows is the full loop, stage by stage.

The thirteen stages

00 · Prereqs — every session, first

The agent checks skills are installed, project files present, version current, no_ui recorded (so frontend checks skip), tests green. Output: pass, in about thirty seconds. The stage's Next block routes a feature request onward to alignment.

01 · Align — grill before code

The first rule fires: this is a feature request, so no code yet. The agent grills until the shape is settled — goal in one sentence ("a coach can see one athlete's progression in one event over time"), history not ranking, oldest-first order, and an explicit out-of-scope list (charts, UI, persistence). The domain word "progression" is standardised to "history view" to avoid two terms.

02 · Context and ADRs — record while it's fresh

CONTEXT.md gains the term history view — a chronological listing of one athlete's results in one event, oldest first, explicitly not the same as the results table. No new ADR: no architectural decision was taken (the existing in-memory-store decision still covers the constraint that matters).

03 · PRD — write down what was agreed

A ~25-line PRD captures the requirements: history(athlete, event) returns results sorted by date ascending; the personal best is identifiable; date ties preserve insertion order; no persistence changes. Success: a coach can answer "is Aisha improving in the 100m?" from one call. State updates to prd-written.

04 · Breakdown — vertical slices

Three slices, each cutting store → query → tests: (1) the store supports per-athlete, per-event chronological retrieval; (2) the history() query with the PB flagged; (3) a formatting helper for display. State: sliced, three slices open.

05 · Triage — pick the next slice

Slice 1 wins — everything depends on it. On a busier project this stage also keeps the tracker honest: dead issues closed, duplicates merged.

06 · Architecture — only when the cadence fires

Not this time: the slices-since-last-review counter is below threshold. The stage exists so the check happens, not so it always runs.

07 · Implementation — TDD

Slice 1, red first:

def test_history_is_chronological_oldest_first(self):
    rows = self.t.history("Aisha", "100m")
    self.assertEqual([r.meet_date for r in rows],
                     sorted(r.meet_date for r in rows))

Then the minimal green implementation, then refactor; the commit is tagged per the project's slice convention. State: two slices open, the architecture counter ticks up, the status block is recomputed.

08 · Review — find what the tests don't

One pass before the PR lands: correctness against the PRD, no invented vocabulary (it says "history view", not "progression log"), and the stable-sort claim actually guaranteed — Python's sorted is stable, noted in the PR description rather than trusted silently.

09 · QA

With no_ui: true and no external dependencies, the verification ladder lets the green unit suite carry this slice — the stage is consulted, not skipped silently. The feature ships when all three slices land.

10 · Ship and deploy — and close the docs

After slice 3 lands, the feature ships and the doc-close ritual fires now, at feature ship — not per slice: a release note is written, the planning folder is moved into archive/, both STATUS.md files are updated, and the state is stamped.

11 · Debug — didn't fire

No defects this pass. When one appears: diagnosis before fix, and the fix itself returns through stage 07 as a TDD slice.

12 · Retro and learn

Ten minutes, honestly answered. Went well: the grilling caught the ranking-vs-history ambiguity before any code. Didn't: slice 1 nearly smuggled in a refactor, split on review. Promoted: "split refactors from feature slices" became a session rule in CLAUDE.md.

What to take from this

  • The artefacts are small. A 25-line PRD, a three-row slice table, a six-line retro. The discipline is in the sequence, not the paperwork.
  • The agent never chose the route. The stage and the per-stage signposts did — that's the V0.2 ergonomics working.
  • Three stages did nothing this pass (architecture, debug, and browser QA) and that's correct: they're checks that can fire, not ceremonies that must.
This example is measured, not aspirational. The mid-flight version of this exact project is preserved as a benchmark fixture, where the harness measures agents navigating it.
