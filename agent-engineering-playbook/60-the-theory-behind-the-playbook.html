The Theory Behind The Playbook

Why the process is shaped the way it is. The companion guide, §50, tells you what to type; this one explains why each stage exists and the thinking behind the skills the agent reaches for — grilling, slicing, test-first building, architecture tune-ups, and the rest — in plain English, no engineering background assumed.

Acknowledgement

This playbook grew out of an exploration of how to turn modern AI-agent skills into a practical engineering process for building real applications. It is especially influenced by Matt Pocock's work on composable engineering skills, including mattpocock/skills and skills.sh, and by Garry Tan's GStack project and garrytan/gstack. This guide adapts those ideas into a human-led, agent-assisted playbook built around explicit alignment, shared language, TDD, review, QA, shipping, and learning loops.

01Why have a process at all?

An AI agent can write code astonishingly fast. The bottleneck was never typing speed — it's making sure the fast output is the right output. Almost every rule in this playbook exists to prevent one of four predictable things that go wrong when you build software with an AI.

The four failure modes, and the practice that heads each one off.

These aren't independent. Skip the grilling and you build the wrong thing well. Skip the shared glossary and the grilling itself gets muddier. Let the structure rot and the test-first building slows to a crawl. The stages are ordered so that each one protects the next — which is the whole reason to follow them in order rather than cherry-pick.

02Align before you code

"No-one knows exactly what they want" is the oldest truth in software. The cheapest moment to fix a misunderstanding is before any code exists. So the process front-loads a conversation whose only job is to make the fuzzy parts of a request explicit.

The agent has a few different grilling tools, each asking a different kind of question:

  • /office-hours the start-up advisor
    A handful of forcing questions that step right back: are we even solving the right problem, and is there a bigger, better version of it hiding inside the request? Strategic framing before anything else.
  • /plan-ceo-review the founder
    Looks at the request the way a founder would — "is there a 10-star product in here?" — then decides whether to expand the scope, hold it, or deliberately cut it back.
  • /grill-with-docs the workhorse
    Interrogates the plan one question at a time, always offering its own recommended answer, and reads the existing code instead of asking you things it can find out for itself. It stops only when every open question has either an answer or an explicit "out of scope".
  • /grill-me for non-code work
    The same discipline for a decision or a strategy doc — anything where there's no codebase to challenge the plan against.
Why "one question at a time, with a recommendation"? It lets you move fast by simply approving the suggested answer when it's right — while guaranteeing that nothing important is ever silently assumed. You stay in control without having to write a specification.

03A shared language

The single cheapest piece of leverage in the whole process: agree on what to call things, write it down, and then use those exact words everywhere.

Concept

— the project's glossary

A living list of the project's real-world nouns, in the words you actually use. Once it exists, the agent uses those same words in the code, the file names, the test descriptions, even the commit messages. That consistency quietly removes a whole category of confusion. It holds domain words only — the language of your world, not technical jargon.

Concept

— recording the "why" behind big decisions

An Architecture Decision Record is a one-page note explaining why a hard-to-undo choice was made. The agent only writes one when the decision passes all three parts of a test: it's hard to reverse, surprising without the back-story, and a genuine trade-off. Otherwise it's just noise. The payoff comes months later, when nobody has to ask "why on earth did we build it this way?"

Both of these are a by-product of the grilling in §02, not extra paperwork — the agent writes to them as decisions settle.

04Slice the work the right way

How you cut a big feature into pieces decides whether you see working software early or only at the very end. The playbook is dogmatic about this: cut vertically, never horizontally.

A : one shot fired all the way to the target, so you can see where it lands and adjust.

Once the plan is settled, /to-prd writes a short spec and /to-issues breaks it into these slices, in the order they depend on each other. Each slice is tagged — whether the agent can finish it alone or needs you in the loop.

Why you should care. Vertical slices mean you get something real to look at early and often, and the agent pauses after each one for your go-ahead. If priorities change, you've committed almost nothing — the opposite of discovering at the end that the whole thing was built around a wrong guess.

05Build it test-first

The agent builds each slice using , which feels backwards at first: you write a small check describing what the code should do before writing the code itself.

The rhythm is three short steps, repeated one tiny piece at a time:

  • . Write a test for the next bit of behaviour. It fails — there's no code yet.
  • Green. Write just enough code to make that test pass. Nothing more.
  • . Tidy the code up while the test keeps passing.

Why bother with the odd order? Because if you write the code first and the tests afterwards, you tend to test what you happened to build, not what you actually wanted. Writing the test first pins down the intended behaviour while it's still clear in everyone's head.

The /tdd skill also enforces a few quiet rules that pay off later:

  • Tests check behaviour through the "front door" (the ), not the internal wiring — so they keep working even when the insides are rewritten.
  • No sneaking in features nobody asked for.
  • The names in the tests use the CONTEXT.md vocabulary, so the shared language reaches all the way down.
The horizontal anti-pattern, again. Writing every test up front and then all the code is the same mistake as horizontal layers from §04 — it tests imagined behaviour. The skill refuses it. One test, one piece of code, repeat.

06Keep the structure healthy

Code that is never tidied rots into a tangle where every change is risky — a . Because agents produce code so fast, the rot is faster too. So structural upkeep is a scheduled stage, not an afterthought.

Think of a car's accelerator: one pedal (a narrow ) hiding a sophisticated engine. That's a deep module.

A handful of structural ideas compound over the life of a project:

  • Deep over shallow modules. A with a small, simple way to use it — hiding the complexity inside — is easier for both you and the agent to use without breaking.
  • One . Each piece of data lives in exactly one canonical place; everything else reads or derives from it. This kills "these two screens disagree" bugs.
  • at the edges. Screens and outside connections just translate to and from that canonical form. The real rules stay in one place, easy to find and change.

/improve-codebase-architecture is the scheduled tune-up — run every few slices, or any time the code "feels heavy". It looks for shallow modules to deepen and tangles to pull apart, and flags anything that contradicts an existing ADR so you can decide deliberately. For a high-stakes structural call, /codex can fetch a second opinion from a different AI model.

Real example

Turning scattered URL rules into one deep module

In one RuChamps architecture pass, the Transport page had a very normal kind of drift: several small files all knew a little bit about how a URL should behave. One helper knew how filters worked. Another knew which drawer parameters to clear. Another knew passenger sorting. The page itself knew about championship-scoped paths. None of those pieces was terrible on its own, but the was effectively "remember all the URL rules correctly every time". That is a shallow module smell. 

Before:
Transport page
  -> filter path helper
  -> drawer path helper
  -> passenger expansion helper
  -> success/error path helper
  -> championship base path rules

After:
Transport page
  -> Transport URL state module
       hides filters, drawers, expansion, notices, dates, scoped paths

The fix was not to add more tiny helpers. It was to deepen the module: give the rest of the code one place to ask "what should the next Transport path be?" and put the fiddly rules behind that single (the clean joining line between two parts of the code). The payoff is everything-in-one-place. If Transport later changes how passenger expansion or championship paths work, the agent has one module to edit and one set of tests to run, instead of rediscovering the rule scattered across the page, forms, drawers, and the code that runs on the server.

07Verify in proportion to the risk

Not every change deserves the same scrutiny. A typo needs a glance; a change to how money or personal data is handled needs tests, a security review, and a careful rollout. The spells out the minimum for each.

"When in doubt, climb the ladder. Over-verifying a small change costs minutes; under-verifying a big one costs weeks."

The skills that do the checking:

  • /qa the QA lead
    Opens a real browser and uses the app like a person would, catching things the automated tests miss — then writes a so that particular bug can never quietly come back.
  • /review the staff engineer
    A careful read for bugs that pass the automated checks but would blow up in real production use.
  • /cso the security officer
    A security review. The rule is absolute: run it on anything touching login, payments, personal data, or any part of the app the outside world can reach — no exceptions.
  • /design-review · /devex-review the specialists
    For the screens your users see, and for the tools other software (or other programmers) plug into — the connection points one program offers another, and text-only tools run by typing commands — respectively.
  • /diagnose the methodical debugger
    When something breaks: reproduce it first, write three-to-five ranked guesses before poking at anything, change one thing at a time, and fix it with a regression test. Its iron law is "no fixes without investigation" — no guessing-and-hoping.

08Ship, then learn from it

The last two stages get the work out the door — and then make the next cycle a little better than this one.

  • /ship the release engineer
    Runs the final checks and opens the for your review. It never merges its own work — you stay accountable for what goes live. (The full hand-off is the subject of §50.)
  • /land-and-deploy · /canary · /benchmark getting it live, safely
    Merge, watch the deployment, keep an eye on it in production (), and measure speed so a slowdown would be noticed rather than discovered by users.
  • /retro · /learn · /document-release closing the loop
    A short look back at what worked and what didn't, captured into the agent's memory () so it improves over time. This is also how a lesson can graduate into the playbook itself.
Tidy on the way out. Shipping also triggers the "doc-close": lasting decisions and vocabulary get promoted into the permanent docs, and finished planning notes are archived. That's what keeps the next session starting from a clean, current picture rather than a pile of stale plans.

09Designing for the agent

The frontend track disciplines what your users see. A newer principle disciplines what the agent sees — because the agent is a reader too, and a slow or cluttered surface costs it time and attention the same way a cluttered screen costs a person.

An agent takes in everything through a narrow window, and it pays — in time, and in how much it can hold in mind at once — for every extra word it has to wade through and every extra back-and-forth it has to make just to learn what to do next. So V0.2 added one rule to the foundations: treat the agent's attention as a first-class design constraint, the way the frontend track treats your visual vocabulary. If a tool, a file, or a document forces the agent to read a wall of text, guess at the next step, or make a second request just to find out "how many?" or "did it pass?", that is a defect — even when it works perfectly for a human.

Where this touches your own product. The moment your product exposes something an agent will use — a tool you run by typing commands, a connection point other software plugs into, or a link-up with another system — the same discipline applies to it: short, tidy output; the few details that matter shown by default; totals already added up; a clear "nothing here" message instead of silence; and a hint about what to do next. The full ten-point checklist lives in the foundations, applied once when the tool is designed and again when it is reviewed.

The playbook now holds itself to that same standard — which is most of what changed between V0.1 and V0.2, under the hood, without altering the thirteen stages themselves:

  • A one-page digest. The agent starts from a compact AGENT-DIGEST.md — the whole map of the playbook in one short read — instead of thousands of lines of prose before it can act. The long guides became reference it opens only when the digest points it there.
  • Signposts at every step. Each stage now ends with a Next: block naming where to go based on the outcome, so the agent walks the process forward instead of returning to a central map after every step.
  • A status line that is already worked out. The project's state file carries a pre-computed summary — what's overdue, which features sit at which stage — so each session reads one answer rather than re-deriving it from two files every time.
What this means for you: nothing changes. You still type the same plain-English requests from §50. These are improvements to how the agent reads its own instructions — quietly making each session faster and cheaper, and less likely to lose the thread.

10Two toolkits, working together

The skills the agent uses come from two open-source collections. You never have to choose between them — the playbook calls the right one at the right moment — but it helps to know why there are two.

Matt Pocock's skills

The fundamentals

  • Small, sharp, single-purpose tools.
  • Cover how to build correctly: grilling, slicing, test-first, deep modules, disciplined debugging.
  • You keep control of the process.

gstack (Garry Tan)

The virtual team

  • Each skill plays a role: founder, designer, QA lead, security officer, release engineer.
  • Cover the whole loop around the build: post-code review, security, deploy, retro.
  • Runs as one work cycle: Think → Plan → Build → Review → Test → Ship → Reflect.

They're complementary, not rivals. Matt's tools are the build; gstack is the loop around the build, filling in the parts Matt deliberately leaves out. This playbook is the stitching that decides which to reach for at each stage.
