The Frontend Track

The extra discipline for anything with a screen. The biggest source of churn in AI-assisted UI work is not bad code — it is the human and the agent using the same word for different things, or the same component for different jobs. Three files prevent that.

The Three Design Artefacts

Three files are the source of truth for what the UI is allowed to look like and how it is allowed to behave. Each answers a different question about the same screen.

The

The textual language. Every UI term — pill, container, primary button — is defined exactly once, so "pill" means the same thing to you, the agent, and the code.

The

The visual language. Every glossary term rendered live on one page that opens in any browser — every approved colour, spacing, and component, with nothing hiding in the code.

The

The interaction language. The kitchen sink shows what a drawer looks like; the guide says when a drawer is the right answer — its decision table maps each job to a .
The hard rule: never invent UI vocabulary or patterns. If a term is missing from the glossary, or a screen needs a surface the guide doesn't sanction, the agent must stop, ask you to decide, and update the artefacts before writing component code. The artefacts lead; the code follows.

Project Setup: The Bootstrap Question

Since V0.1.9, any project with a UI must answer this question at setup. The agent may not silently skip the artefacts, and may not silently impose the playbook defaults — it has to ask:

AgentThis project has a UI. How do you want to set up the three frontend design artefacts (glossary, kitchen sink, design-language guide)?

1 · Use the playbook defaults — I copy all three starter templates, then run the design interview to adapt them to your brand.
2 · Supply your own — point me at existing files, a Figma export, screenshots, or a component library, and I'll distil them into the three-artefact shape.
3 · Mix — defaults for some, your own for others.
4 · Skip for now — only if the project genuinely has no UI. I'll record that decision in so you aren't re-asked every session.
YouUse the defaults — but our brand colour is teal, not gold.
AgentCopying the three templates now, then I'll run the design interview so teal and the rest of your choices land in the artefacts — not just in this chat.

Whatever you answer, the session must end with all three artefacts present in the project — or an explicit recorded skip. Since V0.1.10 the starter kitchen sink is a full operational UI baseline (tables, drawers, overlays, dense app-shell components) rather than a thin sampler, so "use the defaults" gives you a realistic starting vocabulary.

The Design Interview

If no design system exists yet, the agent runs a at kickoff — eleven questions covering colours, type, spacing, radii, default containers, interaction states, the accessibility floor, and (new in V0.1.9) three questions about behaviour: whether screens default to dense rows or cards, which surface handles add/edit/confirm/history, and what viewports the app assumes.

The rule that makes the interview matter: every answer must land in one of the three artefacts — a glossary entry, a kitchen-sink section or CSS token, or a decision-rule row in the guide. No answer is allowed to live only in the conversation transcript.

How The Track Plugs Into The Process Map

The frontend track is not a separate process — it adds one UI check to each stage of the 13-stage map:

  • Prereqs (00). Verify the three artefacts exist. Missing and no recorded skip? Halt and run the bootstrap question.
  • Align (01). After grilling a UI-affecting feature, make or update an HTML using the project's visual language — choosing surfaces by the guide's rules.
  • Context (02). New tokens go into the kitchen sink, new terms into the glossary, new interaction patterns into the guide.
  • Breakdown (04). UI name the glossary terms they use in their acceptance criteria.
  • Architecture (06). Audit for drift: any style in the code that isn't in the kitchen sink is tech debt; any modal doing a drawer's job breaks the guide.
  • Implementation (07). Code uses only glossary terms, only kitchen-sink styles, only sanctioned surfaces.
  • Review (08) & QA (09). Every UI-touching change passes the below.

Keeping Older Projects Current

Also new in V0.1.9: every project records the playbook version it was set up with in . At the start of a session, if the project is behind the playbook's newest version, the agent mentions it and offers , which walks the changelog and applies the differences carefully — new files are copied, boilerplate is patched with the diff shown, and anything you wrote yourself is only ever proposed, never overwritten. It won't force an upgrade mid-feature; it just makes sure improvements like this track actually reach projects that started earlier.
