The Frontend Track

The extra discipline for anything with a screen. The biggest source of wasted, repeated rework in AI-assisted UI (user-interface) work is not bad code — it is the human and the agent using the same word for different things, or the same on-screen building block for different jobs. Three files prevent that.

The Three Design Artefacts

Three files are the single agreed authority for what the screen is allowed to look like and how it is allowed to behave. Each answers a different question about the same screen.

The

The words. Every screen term — pill, container, primary button — is defined exactly once, so "pill" means the same thing to you, the agent, and the code.

The

The visual language. Every glossary term shown live on one page that opens in any browser — every approved colour, spacing, and on-screen building block, with nothing hidden away in the code.

The

How it behaves. The kitchen sink shows what a slide-in drawer looks like; the guide says when a drawer is the right answer — a simple table matching each job to the right (the kind of panel it appears in).
The hard rule: never invent new screen words or patterns. If a term is missing from the glossary, or a screen needs a the guide hasn't approved, the agent must stop, ask you to decide, and update the three files before writing any screen code. The files lead; the code follows.

Project Setup: The Setup Question

Since V0.1.9, any project with a screen must answer this question right at the start. The agent may not quietly skip the three files, and may not quietly force the playbook's default choices on you — it has to ask:

AgentThis project has a screen. How do you want to set up the three frontend design files (glossary, kitchen sink, design-language guide)?

1 · Use the playbook defaults — I copy all three starter templates, then run the design interview to adapt them to your brand.
2 · Supply your own — point me at existing files, a design tool export (such as Figma), screenshots, or a ready-made set of components, and I'll reshape them to fit the three files.
3 · Mix — defaults for some, your own for others.
4 · Skip for now — only if the project genuinely has no screen. I'll record that decision in so you aren't asked again every session.
YouUse the defaults — but our brand colour is teal, not gold.
AgentCopying the three templates now, then I'll run the design interview so teal and the rest of your choices land in the artefacts — not just in this chat.

Whatever you answer, the session must end with all three files present in the project — or a clearly recorded decision to skip. Since V0.1.10 the starter kitchen sink is a full, working set of real screen parts (tables, slide-in panels, pop-ups, and the dense layout of a working app) rather than a thin sample, so "use the defaults" gives you a realistic set of words and parts to start from.

The Design Interview

If no design system (an agreed set of styles and rules for the look) exists yet, the agent runs a at the start — eleven questions covering colours, typefaces, spacing, how rounded the corners are, the default boxes things sit in, how things respond when you hover or click, the accessibility basics, and (new in V0.1.9) three questions about behaviour: whether screens lean towards dense rows or roomier cards, which panel handles add / edit / confirm / history, and which screen sizes the app is built for.

The rule that makes the interview matter: every answer must land in one of the three files — a glossary entry, a section or style value in the kitchen sink, or a decision rule in the guide. No answer is allowed to live only in the chat history.

How The Track Plugs Into The Process Map

The frontend track is not a separate process — it adds one screen-related check to each stage of the 13-stage map:

  • Prerequisites (00). Check the three files exist. Missing, with no recorded decision to skip? Stop and run the setup question.
  • Align (01). After pressure-testing a feature that affects the screen, make or update a clickable (a preview of the screen) using the project's visual language — choosing panels by the guide's rules.
  • Context (02). New style values go into the kitchen sink, new terms into the glossary, new ways of behaving into the guide.
  • Breakdown (04). Each small piece of screen work (a ) names the glossary terms it uses in its pass/fail conditions.
  • Architecture (06). Check for drift: any style in the code that isn't in the kitchen sink is a shortcut to be tidied up later; any pop-up doing a slide-in panel's job breaks the guide.
  • Implementation (07). Code uses only glossary terms, only kitchen-sink styles, only approved panels.
  • Review (08) & QA (09). Every change that touches the screen passes the below.

Keeping Older Projects Current

Also new in V0.1.9: every project records the playbook version it was set up with in . At the start of a session, if the project is behind the playbook's newest version, the agent mentions it and offers , which works through the list of changes and applies them carefully — new files are copied, standard supplied files are updated with the exact changes shown to you, and anything you wrote yourself is only ever suggested for change, never overwritten. It won't force an upgrade in the middle of a feature; it just makes sure improvements like this track actually reach projects that started earlier.
