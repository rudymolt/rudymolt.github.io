The Collaboration Contract

The agent should read context, ask before inventing rules about your world, push back constructively, and avoid work you did not ask for. You own intent and approval; the agent owns execution and verification. You saw this contract in action across the whole of Drive the agent — this page states it as a principle.

Designing for the Agent-Facing Surface

New in V0.2. The frontend track disciplines the surface your users see; this principle disciplines the surface agents consume. It applies whenever the product exposes a CLI, an API, an MCP server, or any tool interface an AI agent will call — increasingly the default, not the exception.

The principle: treat the consuming agent's token budget and turn count as first-class design constraints, the way the frontend track treats visual vocabulary. An interface that forces an agent to take extra turns, parse walls of text, or guess at next steps is a defect, even if it works perfectly for humans.

The checklist — applied when the interface is designed (stage 06) and when it is reviewed (stage 08):

  1. Token-efficient output — compact structured formats; no decorative output.
  2. Minimal default schemas — return the three or four fields agents actually need; offer an escape hatch for the rest.
  3. Content truncation — cap large fields with a size hint and a way to fetch the remainder.
  4. Pre-computed aggregates — include totals, counts, and rolled-up statuses so the agent never needs a second call to know "how many" or "did it pass".
  5. Definitive empty states — print "0 results", never nothing. Silence is indistinguishable from failure.
  6. Structured errors and exit codes — idempotent mutations, machine-parseable errors, no interactive prompts.
  7. Ambient context — let a session start with relevant state already visible rather than requiring discovery calls.
  8. Content first — the no-argument invocation shows live state, not a wall of help text.
  9. Contextual disclosure — append concrete next-step suggestions after output.
  10. Consistent help — every subcommand answers --help concisely, as the fallback.
Litmus test. Could an agent complete the product's three most common workflows without ever reading full documentation — just by following what the interface itself discloses? If not, name the missing disclosure and fix the interface, not the docs. This playbook practises it: is the minimal default schema, the stage signposts are contextual disclosure, and the is a pre-computed aggregate.

The Security Floor

New in V0.2. These are minimums, not a security programme — they exist because their absence hurts without warning. The is checked cheaply every session (prereqs) and per pull request (review).

  • Secrets never enter tracked files. Credentials, tokens, and keys live in environment variables or untracked files covered by .gitignore. An agent that needs a secret stops and asks — it never writes one into code, config, tests, or chat output "temporarily".
  • A committed secret is rotated, not deleted. Removing the line does not remove it from history. Treat it as an incident: rotate first, then clean history if hosting requires it.
  • Dependencies are deliberate. Lockfiles are committed; every new dependency is named and justified in the PR. Prefer the standard library for trivial needs.
  • Validate at every boundary the project owns. User input, file contents, API responses, environment values — validated where they enter, not where they break.
  • Least privilege by default. Tokens and keys get the narrowest scope that works, with an expiry where the platform supports one.
  • Irreversible operations get a rehearsed reverse. Migrations, bulk edits, and deletions are preceded by a timestamped backup, and their rollback has actually been run against a copy before the forward change ships.
