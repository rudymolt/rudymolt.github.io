The Collaboration Contract

The agent should read context, ask before inventing rules about your world, push back constructively, and avoid work you did not ask for. You own intent and approval; the agent owns execution and verification. You saw this contract in action across the whole of Drive the agent — this page states it as a principle.

Designing for the Agent-Facing Surface

New in V0.2. The frontend track is about the surface your users see; this principle is about the surface that other software, including AI agents, uses. It applies whenever your product offers a way to be operated by software rather than by a person clicking around — a command-line tool, an API (a doorway other programs use to talk to it), or any interface an AI agent will call. That is increasingly the norm, not a special case.

The principle: design for the agent's limits the way the frontend track designs for people's. An AI agent can only take in so much at once, and every back-and-forth step costs it time, so an interface that makes it wade through walls of text, take extra steps, or guess what to do next is a fault — even if a person would cope with it fine.

The checklist — applied when the interface is designed (stage 06) and when it is reviewed (stage 08):

  1. Keep output compact — tidy, structured results with no decorative padding the agent has to read past.
  2. Show the essentials by default — the three or four pieces of information the agent actually needs, with a clear way to ask for the rest.
  3. Trim long content — shorten very long results, say how much was left out, and offer a way to fetch the remainder.
  4. Do the sums up front — include totals, counts, and an overall "did it pass?" so the agent never has to ask a second time to learn "how many" or "did it work".
  5. Always say when there is nothing — print "0 results" rather than staying silent, because silence looks exactly like a breakdown.
  6. Make errors clear and safe to repeat — report failures in a fixed, readable form, make running the same step twice harmless, and never stop to ask a question mid-task.
  7. Lead with the current state — show the relevant situation as soon as a session starts, instead of making the agent go hunting for it.
  8. Content first — running the tool with no instructions should show what is going on right now, not a wall of help text.
  9. Suggest the next step — after each result, point to the obvious things the agent might do next.
  10. Consistent help — every command can explain itself briefly when asked, as the fallback.
Litmus test. Could an agent complete the product's three most common workflows without ever reading full documentation — just by following what the interface itself discloses? If not, name the missing disclosure and fix the interface, not the docs. This playbook practises it: is the minimal default schema, the stage signposts are contextual disclosure, and the is a pre-computed aggregate.

The Security Floor

New in V0.2. The is a small set of safety basics that must always be in place — the lowest acceptable bar, not a full security programme. They matter because a missing one rarely looks like a problem at the time: the app keeps working and nothing flags an error, right up until a leaked password or an accidental mass-deletion causes real, sudden damage. So the agent checks the floor cheaply at the start of every session, and again each time it proposes a batch of changes for review (a ).

  • Secrets never enter tracked files. Credentials, tokens, and keys live in environment variables or untracked files covered by .gitignore. An agent that needs a secret stops and asks — it never writes one into code, config, tests, or chat output "temporarily".
  • A committed secret is rotated, not deleted. Removing the line does not remove it from history. Treat it as an incident: rotate first, then clean history if hosting requires it.
  • Dependencies are deliberate. Lockfiles are committed; every new dependency is named and justified in the PR. Prefer the standard library for trivial needs.
  • Validate at every boundary the project owns. User input, file contents, API responses, environment values — validated where they enter, not where they break.
  • Least privilege by default. Tokens and keys get the narrowest scope that works, with an expiry where the platform supports one.
  • Irreversible operations get a rehearsed reverse. Migrations, bulk edits, and deletions are preceded by a timestamped backup, and their rollback has actually been run against a copy before the forward change ships.
