Lite Mode

The full thirteen-stage loop earns its fixed costs on substantial projects. A small tool, a script, or an experiment shouldn't have to pay them — but it shouldn't abandon discipline either. is the subset that keeps the parts that prevent rework, drops the machinery that manages scale, and names the exact moment to graduate.

When lite mode applies

All four of these should be true. If any one is false, run the full bootstrap instead.

  • One human. No second contributor, and none planned.
  • Short-lived. Expected life under about two weeks, or under about three features of work.
  • No real UI. A CLI, a script, or a notebook — not screens.
  • Nothing irreversible. No production data, payments, auth, or migrations at stake.
If in doubt, it doesn't apply. The cost of running the full loop on a small thing is a little paperwork. The cost of running lite mode on something that wasn't small is rework you can't see coming.

What survives — non-negotiable even here

Lite mode drops machinery, never principles. These hold at any size.

  • The first rule. A . Lite alignment before any code, every time — this was the single most-broken discipline in testing, and project size doesn't change that.
  • Lite alignment — five questions, asked and answered in chat before code: what is the goal in one sentence? what are the two or three hard constraints? what is explicitly out of scope? what words does the domain use (and do any clash)? how will we know it works?
  • TDD for anything non-trivial. Red, green, refactor. The verification ladder still scales effort with risk.
  • Review before merge. One pass: correctness, security on any input or output boundary, and "would a stranger understand this?"
  • Foundations still apply. Surgical edits, one direction of data flow, no invented domain rules — it's the stage machinery lite mode drops, not these.

What lite mode drops

  • Files. No state file, no cadences file, no planning/ or archive/ folders, no PRD documents. A single trimmed CLAUDE.md (keep the first rule and constraints; delete what doesn't apply) and — once the project survives more than one session — a minimal CONTEXT.md for vocabulary.
  • Stages. PRD, breakdown, triage, the architecture cadence, browser QA (no UI), the doc-close ritual, and the formal retro. Alignment answers live in the chat and in commit messages.
  • Cadences and state. Nothing is tracked, so nothing is overdue — which is exactly why the graduation tripwire below exists.

The lite loop

Five steps, per piece of work.

  1. Align. The five questions answered — no code before this.
  2. TDD. Failing test → code → green → refactor, per slice of work.
  3. Review. One honest pass before merge.
  4. Ship. Merge, tag or note the version, tell whoever cares.
  5. Ask. "Did I learn something the playbook should know?"

Graduation — the tripwire

Checked at every ship. Graduate to the full playbook the first time any of these happens:

  • A second human joins, or will within a fortnight.
  • The project passes about three shipped features, or about two weeks of life.
  • The same bug class appears twice.
  • A real UI appears — screens, not output.
  • You catch yourself wanting a tracker, a planning doc, or "notes for next session".
  • Anything irreversible enters scope: real users' data, auth, payments, migrations.
How to graduate. Run the full bootstrap, keeping your existing CLAUDE.md and CONTEXT.md as the content the templates merge around. Backfill nothing else — the full loop starts from the next feature, not retroactively. Graduation is one-way: if a graduated project feels heavy, tune the cadences at the retro rather than returning to lite.
