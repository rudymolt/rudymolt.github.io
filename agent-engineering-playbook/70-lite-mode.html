Lite Mode

The full thirteen-stage process is worth its set-up effort on substantial projects. A small tool, a quick script, or an experiment shouldn't have to pay that price — but it shouldn't throw away good habits either. is a stripped-back version that keeps the parts that prevent wasted work, drops the extra steps that only matter at scale, and names the exact moment to switch back to the full process.

When lite mode applies

All four of these should be true. If any one is false, set the project up with the full process instead.

  • One person. Nobody else working on it, and nobody else planned.
  • Short-lived. Expected to last under about two weeks, or under about three features of work.
  • No real interface to look at. A command-line tool, a script, or a quick experiment — not on-screen pages people click through.
  • Nothing you can't undo. No real customer data, payments, log-ins, or changes to how data is stored that would be hard to reverse.
If in doubt, it doesn't apply. The cost of running the full process on a small thing is a little paperwork. The cost of running lite mode on something that turns out not to be small is having to redo work you didn't see coming.

What survives — non-negotiable even here

Lite mode drops extra steps, never principles. These hold at any size.

  • The first rule. A . A quick agreement on the plan before any code, every time — this was the single most-broken habit in testing, and project size doesn't change that.
  • The quick agreement — five questions, asked and answered in the chat before any code: what is the goal in one sentence? what are the two or three things it absolutely must (or must not) do? what are we deliberately leaving out? what words does this subject area use (and do any of them clash)? how will we know it works?
  • Write the test first for anything that isn't trivial. This is : write a check that fails, make it pass, then tidy up. The verification ladder still matches the amount of checking to the amount of risk.
  • Review before you combine the new work into the main project. One read-through: is it correct, is it safe wherever information comes in or goes out, and "would a stranger understand this?"
  • Foundations still apply. Make small, precise changes; keep information flowing in one clear direction; never let the agent make up rules about the subject — lite mode drops the extra stages, not these.

What lite mode drops

  • Files. No progress-tracking file, no schedule-of-checks file, no planning/ or archive/ folders, no written-up requirements documents. Just one trimmed-down CLAUDE.md — the project's standing instructions for the agent (keep the first rule and the key constraints; delete what doesn't apply) — and, once the project lasts more than one working session, a short CONTEXT.md recording the project's shared vocabulary.
  • Stages. The written requirements, the breakdown into pieces, the pick-the-next-task step, the regular architecture check, the in-browser testing (there's no interface to test), the document tidy-up at the end, and the formal look-back review. The answers from the quick agreement live in the chat and in the short notes saved alongside each saved change.
  • Schedules and progress tracking. Nothing is tracked, so nothing can fall behind — which is exactly why the switch-back warning below exists.

The lite loop

Five steps, per piece of work.

  1. Agree the plan. The five questions answered — no code before this.
  2. Test first. Write a failing check → write the code → make it pass → tidy up, for each small piece of work.
  3. Review. One honest read-through before you combine it into the main project.
  4. Release. Combine it in, label or note the version, tell whoever needs to know.
  5. Ask. "Did I learn something the playbook should know?"

Graduating — the warning sign to switch back

Check this every time you release. Move up to the full playbook the first time any of these happens:

  • A second person joins, or will within a fortnight.
  • The project passes about three released features, or about two weeks of life.
  • The same kind of bug appears twice.
  • A real interface appears — pages people click through, not just printed output.
  • You catch yourself wanting a progress tracker, a planning document, or "notes for next session".
  • Something you can't easily undo comes into play: real customers' data, log-ins, payments, or changes to how data is stored.
How to switch back. Set the project up with the full process, keeping your existing CLAUDE.md and CONTEXT.md as the content the new templates wrap around. Don't go back and fill in anything else — the full process starts from the next feature, not by reworking the past. The switch only goes one way: if the full process later feels heavy, ease off the regular checks at the look-back review rather than dropping back to lite mode.
