Open Knowledge Format for AI engineering.

A plain-English guide to Google's OKF: what a knowledge bundle is, why coding agents can use it, and when the convention is worth adopting.

One sentence

OKF is a convention for representing knowledge as plain markdown files with YAML frontmatter, stored in git, readable by both people and AI agents.

The problem OKF is trying to solve

AI agents need context before they can do useful engineering work. That context is usually scattered across docs, code comments, tickets, databases, chat threads, diagrams, and product decisions. The problem is not just finding information. It is giving knowledge a stable shape that a person can review and an agent can navigate.

OKF's answer is deliberately simple: put knowledge in normal files, give each file a small amount of structure, connect related ideas with links, and keep the result under version control.

The format in plain English

An OKF bundle is a directory tree of markdown files. Each file is a concept, such as a table, API, metric, runbook, decision, or reference page. The OKF v0.1 spec keeps conformance intentionally thin: non-reserved markdown files need parseable YAML frontmatter and a non-empty type field.

Bundle

A folder of knowledge files that can be cloned, reviewed, searched, hosted, or passed to an agent.

Concept

One unit of knowledge. For engineering this might be an API, service, domain term, metric, or workflow.

Frontmatter

A YAML block for fields such as type, title, description, tags, and resource.

Body

Normal markdown for the explanation, examples, schemas, caveats, and cited source material.

Links

Ordinary markdown links connect related concepts, making the bundle graph-shaped instead of just folder-shaped.

Index and log

Optional index.md and log.md files help readers discover content and understand what changed.

OKF is the format, not the whole Google toolchain

This is the key distinction. The Google repository includes a spec, an enrichment agent, and a visualizer. Only the first part is OKF. The agent and visualizer are useful examples, but adopting OKF does not mean adopting Gemini, Google ADK, BigQuery, or any specific runtime.

A tiny OKF-style concept

The point is not the exact fields. The point is that a human can read it, an agent can load it, and git can show exactly what changed. 

---
type: API Endpoint
title: Create transport assignment
tags: [transport, scheduling, passengers]
resource: /api/transport/assignments
---

# What this does

Creates a bus assignment for a passenger movement.

# Examples

POST /api/transport/assignments

# Related concepts

- [Transport planner](../systems/transport-planner.md)
- [Assignment conflict rate](../metrics/assignment-conflict-rate.md)

# Citations

1. [Transport API reference](../references/transport-api.md)

What an agent gets from this

The frontmatter gives it a compact label and tags. The body gives it the meaning, examples, and source links. The related concepts tell it what to read next instead of guessing.

Why this is useful for AI engineering

Coding agents such as Claude or Codex work better when the context is explicit, inspectable, and bounded. OKF gives teams a practical pattern for packaging that context without locking it inside a proprietary tool.

Agent-readable context

An agent can start from an index, open only relevant concept files, and follow links when it needs more detail.

Reviewable knowledge

Domain knowledge changes can move through pull requests, diffs, comments, and blame like code changes.

Portable memory

The same bundle can support Codex, Claude, search, docs sites, graph viewers, and future tools.

A practical workflow for an engineering team could look like this:

  1. 1Create a small bundle for a domain area, such as transport, billing, or onboarding.
  2. 2Ask the agent to read the bundle index before it changes code in that area.
  3. 3Let the agent follow links to APIs, metrics, decisions, and known failure modes.
  4. 4Update the bundle when the code or domain model changes, then review that update in git.
Example prompt for Claude or Codex 
Before changing the transport assignment flow, read okf/index.md.
Follow only the concept links that relate to passenger assignment, conflict detection,
and save confirmation. Use those files as the source of truth. If the implementation
changes the domain behavior, update the relevant OKF concept file in the same PR.

Where OKF fits, and where it is overkill

OKF is strongest when knowledge is large, linked, repeated, and useful to machines. It is weaker when a simple README or normal documentation page already does the job.

Good fit

  • Data catalogues with many datasets, tables, metrics, and source references.
  • Agent-readable knowledge bases where context should be navigated, not flattened.
  • Teams that want knowledge updates reviewed through git.
  • Cross-tool or cross-organisation knowledge exchange.

Bad fit

  • A small set of prose-heavy pages where a table of contents is enough.
  • Projects where nobody will maintain the concept files after generation.
  • Cases where a graph view would make a simple domain look more complicated.
  • Governed products that need a mature, widely adopted standard today.

A sensible adoption ladder

Treat OKF as conventions to borrow selectively before treating it as a platform decision.

  1. Borrow the language. Use terms like bundle, concept, index, citation, and progressive disclosure.
  2. Structure important docs. Add lightweight frontmatter to the knowledge files agents read most often.
  3. Create a focused bundle. Start with one domain area rather than trying to document the whole company.
  4. Add checks when useful. Validate required fields, broken links, and citation coverage only when the bundle matters enough.
  5. Add viewers or automation later. Generate graphs, search indexes, or RAG pipelines after the file shape has proved valuable.

The decision test is simple: if the main benefit is that files look tidier, do less. If the benefit is that agents and people can share reliable context through git, OKF is worth considering.

Sources and caveats

This explainer is based on the official Google Cloud OKF repository and a local plain-language interpretation written for this site. The important caveat is status: OKF is currently a v0.1 external draft from a single vendor, so it is best treated as an emerging convention rather than a settled standard.
